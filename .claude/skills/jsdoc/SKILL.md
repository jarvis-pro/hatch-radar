# /jsdoc — 补全导出符号的 JSDoc 注释

为目标文件中**所有导出符号**补写专业 JSDoc 注释。已有注释的符号若缺少 `@param`/`@returns` 或行为说明，也一并补全。

用户参数：`$ARGUMENTS`（可选，目标文件路径；省略时取当前对话中最近讨论的文件）。

---

## 语言检测

默认用**中文**撰写 JSDoc。仅当文件内已有的行内注释（`//` 和 `/* */`）多数为英文时，改用英文。

## 需要文档化的符号

- **函数 / 方法**：摘要行 + `@param` 每个参数 + `@returns` 返回值 + 关键行为注解
- **类型 / 接口**：摘要行 + **每个字段**补充行内 `/** */` 注释，描述语义含义与约束（这是 IDE hover 的来源，字段名+类型已足够自明的仍需写，除非完全无歧义）
- **类**：类级摘要 + 构造器参数 + 公有方法按函数规则处理
- **导出常量**：用途与取值约束
- **枚举**：枚举级摘要行，**且每个枚举项单独一条 `/** ... \*/` 行内注释\*\*，精确描述该值的含义与适用场景（消费方选值时应能直接从悬浮提示作出判断，无需查源码）

## 枚举项注释格式

每个枚举项**必须**在其声明行的正上方或行内加 `/** ... */` 注释，即使枚举名已经较为直白。格式优先行内（值短时）：

```ts
export enum ConversationStatus {
  /** 等待坐席接入，尚未有人回复 */
  Pending = "pending",
  /** 坐席已接入，正在进行中 */
  Active = "active",
  /** 会话已正常结束 */
  Closed = "closed",
}
```

若枚举注释较长，改为上方独占行的 `/** */` 块。**不允许**仅在枚举类型级别写一条总注释而跳过各成员。

## 跳过以下情形

- 未导出的内部符号（TSX 头部声明除外，见下节）
- 参数含义已完全由名称+类型捕获，无需补充（如 `id: number` 不需要"the ID"）
- 零逻辑的纯 getter/setter
- 枚举项的名称+值组合已无歧义且语义完全自明（如 `True = true` / `False = false`），但此类情形极少，默认仍写注释

## TSX 组件函数体头部注释（.tsx 文件专属）

当目标文件为 `.tsx` 时，**除常规 JSDoc 外**，还需对每个导出组件函数体的**头部声明区**（`return` 语句之前的所有顶层 `const`/`let` 声明与裸表达式调用）补写 `//` 行内注释。

### 注释对象

return 前的每一条顶层声明 / 调用，包括但不限于：

- `useState`、`useReducer` — 描述**状态含义**，不重复"useState"字样
- `useEffect`、`useLayoutEffect` — 描述**副作用目的**，呼应依赖数组含义
- `useQuery`、`useMutation` — 描述**查询 / 操作的业务语义**
- `useForm`、`useController` — 描述**表单所管理的数据对象**
- `useNavigate`、`useLocation`、`useParams` — 描述**取来做什么用**
- 自定义 hook — 描述业务语义，不复述 hook 名
- 普通 `const` 赋值（从 hook 数据派生的中间变量）
- 内部 handler / helper 函数声明（`function` 或 `const fn = ...`）

### 注释格式

注释放在声明行**正上方**，使用 `//`，一行，简短描述用途；不重复 hook 名，不复述变量名：

```tsx
// 控制弹窗开关
const [open, setOpen] = useState(false)
// 获取详情数据
const { data: detail } = useQuery(...)
// id 变化时重置表单
useEffect(() => { ... }, [id])
// 提交更新
const { mutate } = useMutation(...)
// 提交时阻止默认事件并校验
function handleSubmit(e: FormEvent) { ... }
```

### 跳过条件（头部 // 注释）

- 声明行正上方的紧邻非空行已是 `//` 或 `/* */` 注释（不重复添加）
- 该声明位于 `useEffect`/`useMemo`/`useCallback` 等回调内部（非组件顶层）
- `return` 语句之后的 JSX 内变量

## 注释质量标准

**摘要行**：说明函数*做什么*，而非*怎么做*。一句话，不重复函数名。

**@param**：只描述类型签名之外的信息——约束、默认语义、为空/零时的行为。

**@returns**：说明返回值*代表什么*，以及边界情形（如"无匹配时返回原列表引用"）。

**行为注解**：调用方需要知道的隐含契约——是否修改入参、是否幂等、乐观更新占位替换逻辑、排序稳定性、性能特征等。放在摘要行之后、`@param` 之前，以 `-` 列表呈现（仅在有实质信息时添加，不凑字数）。

**不写**：重述 TypeScript 类型的散文描述；"此函数用于……"套话；对调用方无意义的实现细节。

## 类型字段 vs 实现对象成员：分工，不重复

TypeScript 的 IDE hover 从**类型定义**取注释，不从实现对象字面量取。当存在「类型声明 + 对应实现对象」的成对结构时（例如状态管理 store 工厂、配置对象、用 `satisfies`/类型断言的对象字面量等），两处注释面向不同受众，必须分工：

| 位置                      | 受众               | 写什么                                 |
| ------------------------- | ------------------ | -------------------------------------- |
| `type` / `interface` 字段 | 消费方 hover       | 语义含义、类型约束（"是什么"）         |
| 实现对象的成员            | 阅读该实现源码的人 | 类型注释未覆盖的行为信息（"怎么运作"） |

**规则：**

- `type`/`interface` 字段：**一律**写行内 `/** */`（这是 hover 来源）
- 实现对象成员：**只在有额外行为信息时**才写注释（初始化来源、副作用、边界行为）；若与类型字段注释完全重叠，**省略**，避免逐字复制

下例以工厂函数构造的实现对象为例（任何「类型 + 实现」成对结构同理）：

```ts
type AuthState = {
  /** 当前登录 token；未登录时为 null */
  token: string | null;
  /** 用账号密码登录，成功后持久化 token */
  login: (account: string, password: string) => Promise<void>;
};

export const auth = createState<AuthState>((set, get) => ({
  /** 初始值从持久化存储读取，无值时为 null */  // ← 补充了类型注释没有的初始化信息
  token: initialToken,
  user: null,   // ← 类型注释已足够，省略
  /** 失败时恢复前一认证状态并重新抛出 */  // ← 补充了行为细节
  async login(account, password) { ... },
}));
```

## 执行流程

1. 读取完整文件（`Read` 工具）
2. 枚举所有 `export` 符号，按出现顺序列出待文档化清单
   2b. **若文件为 .tsx**：扫描每个导出组件函数体（PascalCase 函数名），识别头部声明区（开括号 `{` 至第一个顶层 `return` 之间的顶层语句）；跳过正上方紧邻非空行已有注释的声明，列出待注释清单
3. 为每个符号起草 JSDoc 块，直接放在声明上方
4. 用 `Edit` 工具逐个写入（每次 edit 对准一个符号，避免大块替换出错）
   4b. **（.tsx 文件）** 对头部声明待注释清单，用 `Edit` 逐条在声明行正上方插入 `//` 注释行；每次 Edit 对准一条声明，避免大块替换出错
5. 简要列出已文档化的符号名，供用户核查覆盖范围

完成后不做额外总结，只报符号清单。
